## Challenges

### 1. Scope Evolution — Work Done Then Undone
Story 6.2 (PreFilledContext) was fully implemented with 25 tests, then its component was removed when 6.4's DiagramControls provided better inline stats. This represents wasted effort. The spike should ideally have happened BEFORE Story 6.2 implementation.

### 2. Incorrect Clustering Assumption (SIGNIFICANT DISCOVERY)
The pump clustering logic groups by `pumpType` (pump model), merging analyses with the same model. This is physically incorrect — each process corresponds to a distinct location in the fab. Two groups of the same pump model on different processes need separate clusters and separate Micro-PCs.

### 3. Test Selector Regressions (RECURRING)
Story 6.4: `getByText('20')` broke when DiagramControls and PreFilledContext both displayed the same total pumps value. Same pattern as Epic 4 (stories 4.1/4.2 with `getByText(/%/)`). The `within()` convention from Epic 4 retro was not adopted.

### 4. Action Item Follow-Through Degraded
40% follow-through on Epic 4 retro action items (down from 86% on Epic 3 items). Notably, CLAUDE.md test count not updated (603 vs actual 1137) and `within()` queries not adopted.

---

## Key Insights

1. **Spikes before implementation prevent waste** — If the spike had happened before Story 6.2, we would have avoided building and then removing PreFilledContext. Sequence: spike first when scope is uncertain.
2. **Clustering must reflect physical reality** — The fab layout (process = location) dictates infrastructure topology, not pump model similarity. 1 cluster = 1 analysis = 1 process.
3. **Action items need a forcing function** — Living in a retro document is not enough. Must be verified at epic start as a mandatory checklist.
4. **Natural absorption beats forced modularity** — Story 6.4 absorbing 6.2 and 6.3 produced simpler, better architecture than the original 3-story plan.
5. **Prototypes validate faster than specs** — The HTML prototypes from the spike gave the PO something to react to immediately, leading to faster alignment than written specifications.

## Significant Discovery Alert

### Pump Clustering Must Be By Process, Not By Model

**Current behavior (INCORRECT):**
- `usePumpStats` groups analyses by `pumpType`
- 3 analyses with 2 pump models → 2 clusters
- Example: "Poly Etch" (8x A3004XN) + "CVD" (6x A3004XN) = 1 merged cluster "A3004XN: 14 pumps"

**Correct behavior (PO confirmed):**
- Clustering by analysis (= process = physical location)
- 3 analyses → 3 clusters, regardless of pump model
- Example: Cluster "Poly Etch" (8x A3004XN) + Cluster "CVD" (6x A3004XN) = 2 separate clusters

**Why:** Each process corresponds to a specific physical location in the fab. Pumps on different processes are not co-located. Each cluster needs its own Micro-PC in Pilot mode because cabling must stay short.

**Impact:** Story 6-6 to be created. Changes: `usePumpStats.ts`, `PumpCluster.tsx`, `ArchitectureDiagram.tsx`, `ArchitectureDiagram.test.tsx`.

**Rule confirmed by PO:** 1 cluster = 1 analyse = 1 process = 1 lieu physique
